**Technologie-Stack:** C# / .NET 8.0 / ASP.NET Core Minimal API

Keine der verfuegbaren globalen Architekturen passt zu diesem Projekt:
- **dotnet-avalonia-clean-architecture**: Fuer Desktop-Apps mit Avalonia UI, nicht fuer Web API + Console Apps

**Erkannte Architektur:** Hybrid aus Plugin-Architektur + Pipeline Pattern + Service Provider Pattern in einer .NET 8.0 Minimal API + Console Application Struktur. Kein Clean Architecture im strengen Sinn (keine explizite Dependency Inversion mit Outer/Inner Rings).

**Empfehlung:** Die aktuelle Architektur koennte als neue globale Vorlage "dotnet-minimal-api-console-architecture" extrahiert werden, falls dieses Pattern in weiteren Projekten verwendet wird.

## Phase 4: Verbesserungsvorschlaege

### Vorschlag 1: Kritische Command Injection schliessen

**Kategorie:** Sicherheit
**Prioritaet:** Hoch
**Aufwand:** Klein

**Ist-Zustand:** `ScriptExecutor.cs` filtert nur `;` aus Parametern. PowerShell-Injection via `|`, `&&`, `$()` moeglich. `ScriptsProcessor.CheckIfScriptExists()` gibt immer `true` zurueck (prueft nur ob Verzeichnis Dateien hat, nicht ob der angefragte Script existiert).

**Soll-Zustand:** Vollstaendiger Injection-Schutz. Script-Name wird gegen vorhandene Dateien validiert.

**Umsetzung:**
1. `ScriptExecutor.cs:32-40`: `ProcessStartInfo.ArgumentList` statt `Arguments` verwenden (verhindert Shell-Interpretation)
2. `ScriptsProcessor.cs:7-13`: `CheckIfScriptExists` reparieren -- tatsaechlich pruefen ob `Path.Combine(scriptsPath, scriptName)` existiert
3. Whitelist erlaubter Script-Erweiterungen (.ps1, .py)
4. `EmitterOperatingMode.cs:84-94`: Command-Validierung hinzufuegen

---

### Vorschlag 2: Authentifizierung haerten

**Kategorie:** Sicherheit
**Prioritaet:** Hoch
**Aufwand:** Mittel

**Ist-Zustand:** Default-Key "1234", Key in URL-Parametern (in Logs/Proxies sichtbar), einfacher String-Vergleich, Key in Plaintext geloggt.

**Soll-Zustand:** Sicheres Key-Management, Key nicht in URLs oder Logs sichtbar.

**Umsetzung:**
1. Default-Key entfernen -- Server muss mit explizitem Key gestartet werden
2. Key von URL-Parameter auf `Authorization`-Header umstellen (alle Clients + Server)
3. `KeyBasedAccessControl.cs`: Constant-Time-Vergleich verwenden (`CryptographicOperations.FixedTimeEquals`)
4. Server-Logging: Key-Wert nie loggen, nur ob Authentifizierung erfolgreich/fehlgeschlagen
5. Environment-Variable `LNAC_KEY` als alternative Key-Quelle unterstuetzen

---

### Vorschlag 3: Path-Traversal in File-Storage beheben

**Kategorie:** Sicherheit
**Prioritaet:** Hoch
**Aufwand:** Klein

**Ist-Zustand:** `Util.SanatizeFilename()` entfernt nur ungueltige Zeichen. `StorageServiceProvider` validiert nicht, ob der resultierende Pfad innerhalb des Storage-Verzeichnisses bleibt.

**Soll-Zustand:** Dateioperationen sind auf das Storage-Verzeichnis beschraenkt.

**Umsetzung:**
1. `StorageServiceProvider.cs`: Nach `Path.Combine()` mit `Path.GetFullPath()` aufloesen und pruefen ob Pfad mit `_dataPath` beginnt
2. `Util.cs`: `SanatizeFilename` durch `Path.GetFileName()` ersetzen (entfernt alle Verzeichniskomponenten)
3. Typo korrigieren: `Sanatize` -> `Sanitize`
4. File-Upload-Groessenlimit einfuehren (z.B. 100 MB)

---

### Vorschlag 4: Rate Limiting und HTTPS-Default einfuehren

**Kategorie:** Sicherheit
**Prioritaet:** Hoch
**Aufwand:** Mittel

**Ist-Zustand:** Kein Rate Limiting auf Endpoints. HTTP ist Default, HTTPS optional. Keine Security-Header.

**Soll-Zustand:** Brute-Force-Schutz, verschluesselte Kommunikation als Default.

**Umsetzung:**
1. `Microsoft.AspNetCore.RateLimiting` NuGet-Paket einbinden
2. Fixed-Window-Rate-Limiter: 100 Requests/Minute pro IP
3. Auth-Failure-Logging mit Rate-Limit-Info
4. HTTPS als Default, HTTP nur mit explizitem `--no-https` Flag
5. Security-Header hinzufuegen: X-Content-Type-Options, X-Frame-Options

### Strategische Einordnung

Die Relevanz-Bewertung ergibt in allen drei Dimensionen 🔴 (Abloesen). Das bedeutet nicht, dass LNAC wertlos ist -- es loest ein reales Problem und demonstriert solide Architektur-Patterns. Aber der Markt hat sich seit 2022 fundamental veraendert:

- **Als Chat-Tool** wird LNAC von Mattermost, Zulip, Rocket.Chat komplett uebertrumpft
- **Als Automation-Tool** steht es gegen Ansible, Rundeck, n8n, die alle ausgereifter sind
- **Als Messaging-Layer** ist NATS (19.5k Stars, CNCF-Projekt) die ueberlegene Basis
- **Als Bot-Pattern** werden AI-Agent-Frameworks (LangGraph, CrewAI, n8n AI Agents) den Use-Case subsumieren

**Gesamtempfehlung:**

Falls das Projekt weiterentwickelt wird, sollten die Sicherheits-Fixes (Vorschlaege 1-4) **sofort** umgesetzt werden -- unabhaengig von der strategischen Entscheidung. Ein unsicheres Tool sollte nicht im Netz laufen.

Fuer die strategische Ausrichtung gibt es drei sinnvolle Pfade:
1. **Lern-/Portfolio-Projekt**: Explizit als solches positionieren. Die Architektur-Patterns (Pipeline, Plugin, Strategy) sind lehrreich und gut umgesetzt.
2. **Nische besetzen**: "Zero-Config .NET LAN-Automation" -- der einzige Raum wo LNAC sich differenzieren koennte.
3. **Migration**: Use-Cases auf NATS (Messaging) + Rundeck/n8n (Automation) migrieren und LNAC in Wartungsmodus setzen.
